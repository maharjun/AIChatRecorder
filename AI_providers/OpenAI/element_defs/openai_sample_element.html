You said:
Is there a programmatic way to check using javascript whether a particular file extension corresponds to a text file or a binary file?

There isn’t a built‐in JavaScript API that “magically” knows if an extension is text or binary. In practice, you have two common approaches:

  1. Heuristic based on MIME type:
    You can use a library (like mime-types) to map a file extension to its MIME type. Then, if the MIME type starts with "text/" or is one of a few known text-based types (like JSON or XML), you can assume it’s a text file. For example:

    javascript
    const mime = require('mime-types');

const ext = '.txt';
const mimeType = mime.lookup(ext);

if (mimeType && mimeType.startsWith('text/')) {
  console.log('Text file');
} else {
  console.log('Likely a binary file');
}

    This approach works well for many cases, but it relies on conventions and isn’t foolproof—file extensions don’t always match content.

  2. Content inspection:
    Sometimes you might want to inspect the file’s actual bytes. Libraries like istextorbinary can help determine whether a file’s content is text or binary by analyzing its content rather than just its extension. For instance: 

    javascript
    const fs = require('fs');
const { isText } = require('istextorbinary');

fs.readFile('path/to/file', (err, data) => {
  if (err) throw err;
  const result = isText(null, data); // or use isBinary() similarly
  console.log(result ? 'Text file' : 'Binary file');
});

Each method has its trade-offs. Relying solely on the file extension (the MIME type approach) is simple but can be inaccurate if the extension is misleading. On the other hand, analyzing file content gives a more accurate result, but it requires reading the file data, which might not always be feasible (for example, in browser environments).

In summary, while there isn’t a one-line native function in JavaScript to check if a file is text or binary based on its extension, you can combine MIME type libraries or content-inspection libraries to implement this functionality.

You said:
I'd like to do this purely with the name of the file in the chrome browser. The image is likely irrelevant

If you only have access to the filename in the browser (no file contents and no server‐side libraries), the simplest approach is to maintain a list (or map) of “known” text‐based extensions and check against it. For example:

js
function isLikelyTextFile(filename) {
  // Define a list of extensions commonly used for text files
  const textExtensions = [
    'txt', 'md', 'json', 'js', 'ts', 'css', 'html', 'htm',
    'xml', 'csv', 'yml', 'yaml', 'sh', 'py', 'java', 'c',
    'cpp', 'cs', 'php', 'rb', 'env' // and so on...
  ];
  
  // Extract the part after the last dot
  const ext = filename.split('.').pop().toLowerCase();
  
  return textExtensions.includes(ext);
}

// Example usage:
console.log(isLikelyTextFile('document.txt'));   // true
console.log(isLikelyTextFile('photo.png'));      // false

Caveats

  1. It’s only a guess. File extensions aren’t guaranteed to reflect a file’s true content or format. For instance, a file named script.txt might actually contain binary data.

  2. Limited to known extensions. You have to keep your list of “text” extensions updated if you want to support additional or unusual text formats.

  3. No built‐in MIME type lookup in the browser. Unlike in Node.js (where you can use libraries like mime-types), in a plain browser environment you either maintain your own extension‐to‐MIME map or rely on a server response. If you are using <input type="file" /> to select files, you can sometimes read the file’s MIME type from the File object (e.g., file.type), but that also depends on the OS/browser’s best guess and may not always be accurate.

In other words, if you only have the filename, a simple list‐based check is about the best you can do purely in client‐side JavaScript.
